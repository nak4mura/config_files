---
name: extract_requirements
description: プロジェクトから業務要件を抽出して一覧化
---

プロジェクトの複数のソースから業務要件を抽出し、機能タイプ別に整理してMarkdown形式で一覧化する手順：

## 解析フロー

1. **プロジェクト構造の把握**
   - ルートディレクトリの構造を確認
   - 主要なディレクトリ（docs/, src/, tests/, etc.）を特定
   - 使用言語・フレームワークを特定

2. **ドキュメントからの抽出**
   - ドキュメントファイルを特定（README.md, docs/, requirements/, *.md）
   - 業務要件の記述を抽出：
     - ユーザーストーリー
     - 機能要件
     - 非機能要件
     - 制約条件
     - ビジネスルール
   - キーワード検索：「要件」「仕様」「仕様書」「機能」「仕様」「機能要件」「必須」「〜でなければならない」「〜すること」

3. **テストコードからの抽出**
   - テストファイルを特定（tests/, __tests__/, spec/, test/, *.test.*, *.spec.*）
   - 検証ロジックから制約を抽出：
     - 入力値の制約（必須、範囲、形式など）
     - 出力値の期待値
     - エッジケースのテスト
   - テストケースから期待される振る舞いを抽出：
     - 正常系
     - 異常系
     - 境界値テスト

4. **API 仕様からの抽出**
   - ルーティング定義ファイルを特定（routes/, api/, controllers/, handlers/, etc.）
   - エンドポイントから機能要件を抽出：
     - HTTP メソッドとパス
     - リクエストパラメータの制約
     - レスポンスの構造
     - エラーハンドリング

5. **DB スキーマからの抽出**
   - データベース定義ファイルを特定（migrations/, schema.sql, prisma/schema.prisma, etc.）
   - データ制約を抽出：
     - NOT NULL, UNIQUE, CHECK 制約
     - 外部キー制約
     - デフォルト値
   - リレーションからドメインルールを抽出：
     - 一対一、一対多、多対多の関係
     - カスケード操作

6. **コード内のドキュメントからの抽出**
   - JSDoc、docstring、コメントから要件を抽出
   - 検証ロジック（バリデーション）から制約を抽出
   - エラーハンドリングから制約を抽出

7. **UI 定義からの抽出**
   - UI コンポーネント定義を特定（components/, views/, pages/, etc.）
   - フォームのバリデーションから要件を抽出
   - 画面遷移からユースケースを抽出
   - 表示制御（権限、条件分岐など）から要件を抽出

8. **Issue/PR からの抽出**
   - GitHub Issues を検索（`gh issue list`）
   - 機能追加の提案を抽出
   - ユーザーからの要求を抽出
   - タイトル、本文、コメントから要件を特定

9. **機能タイプ別の整理**
   抽出した要件を以下の機能タイプに分類：

   - **認証・認可**
     - ログイン、ログアウト
     - ユーザー登録、削除
     - パスワードリセット、変更
     - 権限管理、ロール管理
     - セッション管理

   - **データ入力・保存**
     - 作成（新規登録）
     - 更新（修正）
     - 削除
     - バリデーション
     - アップロード

   - **検索・取得**
     - 一覧表示
     - 詳細表示
     - フィルタリング
     - ソート
     - ページネーション

   - **レポート・出力**
     - エクスポート（CSV, Excel, JSON など）
     - PDF 出力
     - 集計
     - グラフ表示
     - 印刷

   - **通知**
     - メール通知
     - プッシュ通知
     - SMS 通知
     - アラート
     - リマインダー

   - **設定・管理**
     - システム設定
     - ユーザー管理
     - マスタ管理
     - ログ管理
     - 監査ログ

   - **データ同期**
     - 外部連携
     - インポート
     - バッチ処理
     - API 連携
     - データ移行

   - **その他**
     - その他の機能タイプ

10. **Markdown 出力の生成**
    以下の構造で業務要件一覧を出力する：

```markdown
# 業務要件一覧

生成日: YYYY-MM-DD

## 認証・認可

### ユーザー登録
- **説明**: 新規ユーザーを登録する
- **詳細要件**:
  - メールアドレスは必須、一意であること
  - パスワードは8文字以上で、英数字を含むこと
  - 登録時に確認メールを送信すること
- **出典**:
  - docs/requirements.md:10
  - tests/auth.test.ts:25
  - src/api/routes/auth.ts:45
- **関連**:
  - API: POST /api/auth/register
  - DB: users テーブル
  - UI: 登録フォーム
  - Issue: #123

### ログイン
...

## データ入力・保存

### データ作成
...

## 検索・取得

### データ一覧表示
...

## レポート・出力

### データエクスポート
...

## 通知

### メール通知
...

## 設定・管理

### システム設定
...

## データ同期

### 外部連携
...

## その他

### その他機能
...

## 統計情報

| 機能タイプ | 要件数 |
|-----------|--------|
| 認証・認可 | 5 |
| データ入力・保存 | 12 |
| 検索・取得 | 8 |
| レポート・出力 | 3 |
| 通知 | 4 |
| 設定・管理 | 6 |
| データ同期 | 2 |
| その他 | 5 |
| **合計** | **45** |

## 未分類の要件

抽出したが分類できなかった要件一覧（あれば記述）
```

## 注意点

- 技術的制約と業務要件の区別はせず、業務要件の可能性があるものはすべて含める
- 出典を明記することで、要件の根拠を追跡可能にする
- 関連する API、DB、UI、Issue をリンクする
- 重複する要件は統合し、出典をすべて記載する
- 出力先は `docs/requirements.md` とする
- 要件が見つからない場合は、その旨を明記する
